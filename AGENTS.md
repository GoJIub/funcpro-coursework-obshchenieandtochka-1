# AGENTS.md

## Проект

Мы реализуем небольшой функциональный язык программирования на F#.

## Главная цель

Построить Lisp-like функциональный язык с интерпретатором, лексическими
замыканиями, рекурсией, списками, явной ленивостью, примерами, тестами и
документацией.

Не генерировать несвязанную полную реализацию языка. Следовать существующей
архитектуре и держать изменения в границах текущего Issue.

## Архитектура

- AST определяется в `src/Language/Ast.fs`.
- Runtime-значения определяются в `src/Language/Values.fs`.
- Ошибки вычисления определяются в `src/Language/EvalError.fs`.
- Интерпретатор основан на окружении и возвращает `Result<Value, EvalError>`.
- Парсер читает Lisp-like S-выражения и возвращает `Result<Expr, ParseError>`.

## Зоны ответственности

- Участник 1 отвечает за семантическое ядро: AST, значения, окружение,
  интерпретатор, замыкания, `letrec`, рекурсию и ошибки выполнения.
- Участник 2 отвечает за парсер, синтаксис, pretty-printer, CLI и ошибки
  парсинга.
- Участник 3 отвечает за встроенные функции, списки, примеры с функциями
  высшего порядка, документацию, логи использования ИИ, GitHub Pages и вывод
  trace.

Не менять общие контракты `Expr`, `Value`, `Env`, `ParseError`, `EvalError`,
сигнатуры парсера/интерпретатора, формат встроенных функций и trace-событий без
отдельного Issue и Pull Request.

## Правила кода

- Предпочитать иммутабельные структуры данных.
- Использовать pattern matching.
- Избегать скрытой мутации, кроме случаев, когда она нужна для memoization
  thunk-ов.
- Не выбрасывать исключения для обычных ошибок языка.
- Возвращать `Result` для ошибок парсера и интерпретатора.
- Добавлять сфокусированные тесты для каждой семантической возможности.
- Делать маленькие Pull Request, привязанные к одному Issue.

## Команды

Сборка:

```bash
dotnet build
```

Запуск тестов:

```bash
dotnet test
```

Запуск CLI-заглушки:

```bash
dotnet run --project src/Cli
```
